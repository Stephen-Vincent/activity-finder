# 2026-05-04 — Schema: tables, RLS, and RPCs

**Duration:** ~2 hours (one sitting)
**Where I started:** Monorepo scaffold on `main`. `supabase/migrations/0001_init.sql` was a TODO comment with no DDL. `DATA_MODEL.md` was a placeholder. The mobile app couldn't talk to a database because there was no database to talk to.
**Where I ended:** Local Supabase running in Docker with the full schema applied. TypeScript types generated and committed. The mobile app's Supabase client is wired up and typechecks. Ready to start building the sign-in screen next session.

## Goal

Take `0001_init.sql` from a TODO comment to a real, applyable initial schema covering everything in [DESIGN.md §7](../DESIGN.md#7-data-model) — tables, indexes, RPCs, RLS — and update `DATA_MODEL.md` to reflect what got built.

## What got done

**The initial migration is written.** [`supabase/migrations/0001_init.sql`](../../supabase/migrations/0001_init.sql) is now a ~700-line single file that the TODO comment promised. It:

- enables `pgcrypto`, `citext`, and `postgis`,
- defines 15 enums (`country_code`, `currency`, `price_band`, `indoor_outdoor`, `venue_status`, `venue_tier`, `plan_type`, `subscription_status`, plus the supporting cast for parking, review status, promotion status, booking provider, school region, distance unit, owner role),
- creates all 19 tables in dependency order (`users`, `staff`, `children`, `categories`, `venues`, `venue_categories`, `venue_owners`, `venue_photos`, `opening_hours`, `events`, `reviews`, `review_photos`, `favourites`, `plans`, `plan_items`, `subscriptions`, `bookings_clicks`, `promotions`, `school_terms`),
- adds the indexes the TODO listed (GiST on `venues.location`, btree on `country_code`/`tier`/`status`, descending composite on `reviews(venue_id, created_at)`, unique composite PKs on `favourites` and `venue_categories`, plus the reverse-join indexes you'll want for `events`, `venue_photos`, `opening_hours`, `bookings_clicks`),
- creates `is_admin()` as a security-definer function so RLS policies can call it without granting `select` on `staff`,
- creates `handle_new_user()` to mirror `auth.users` rows into `public.users` on signup,
- creates the two RPCs: `venues_within(centre_lng, centre_lat, radius_m, filters jsonb, page_size, page_offset)` and `claim_venue(p_venue_id, p_evidence)`,
- enables RLS on every user-touchable table with explicit allow policies — owner-only on `children`/`favourites`/`plans`/`plan_items`, public-read with verified-owner-write on `venues` and its child tables, public-read of approved reviews with author and venue-owner visibility into pending ones, admin-write on reference tables (`categories`, `school_terms`).

**`DATA_MODEL.md` is no longer a stub.** [`docs/DATA_MODEL.md`](../DATA_MODEL.md) now has per-table column references, the enum list, an index summary, an RLS policy summary, the RPC contracts (with the supported `filters` jsonb keys for `venues_within`), and a Mermaid ER diagram. It's the doc you read when the question is "what does table X look like" — the SQL stays the source of truth, the doc explains it.

**Verification done before declaring victory.** Parsed the whole migration through `libpg_query` (the parser Postgres itself uses) — 143 statements, no syntax errors. Then a structural sweep confirmed:
- every `references public.X` target is a table that gets created,
- every table has RLS enabled and at least one policy attached,
- every `set_updated_at` trigger points at a table that actually has an `updated_at` column.
This catches structural drift but not full semantics — the real test is `supabase db reset`.

**Small drift fix in the seed README.** The CSV at `supabase/seed/venues.sample.csv` doesn't include a `currency` column, but the schema requires `venues.currency NOT NULL`. Added a note to [`supabase/seed/README.md`](../../supabase/seed/README.md) telling whoever implements `seed-venues.ts` to derive currency from `country_code` (`GB-NIR`→`GBP`, `IE`→`EUR`), semicolon-split the categories field, and slug-fallback when the row has no slug.

**Bringing up Supabase locally surfaced two install-time gotchas:**

1. The Supabase CLI wasn't installed at all. Decided to add it as a dev dependency (`npm install --save-dev supabase`) rather than a global Homebrew install, so `npm install` is enough to onboard a future contributor at the right CLI version.
2. The first `supabase start` failed parsing `supabase/config.toml`: `'functions[verify_jwt]' expected a map or struct, got "bool"`. The CLI now expects one `[functions.<name>]` block per function rather than a single `[functions] verify_jwt = …`. Rewrote the block with one entry per function, `verify_jwt = false` for the webhook/redirect/scheduled jobs and `true` only for `create-subscription` (which is called by the authenticated mobile app).

**Migration applied green on first real run.** Once the config was fixed, `supabase start` came up cleanly and applied `0001_init.sql` automatically. Two messages came back labelled "NOTICE" — both are just informational, not errors:

- "extension pgcrypto already exists, skipping" — Supabase pre-installs that extension; our migration politely tried again and was waved through.
- "trigger on_auth_user_created … does not exist, skipping" — we wrote a defensive "delete this trigger if it's there" line so the migration can be re-run safely. On a brand-new database there was nothing to delete, which is fine.

One small WARN: "no files matched pattern: supabase/seed.sql". The CLI looks for a single seed file by that exact name; ours are split into `supabase/seed/categories.sql` and `seed/school_terms.sql`. Easy fix when those files actually have content — point the CLI at the folder via `[db.seed]` in `config.toml`.

**Studio eyeball check passed.** Opened <http://127.0.0.1:54323> → Table Editor and confirmed all 19 public-schema tables are present with the right columns and enums. The schema is real.

**Heads-up about Supabase's new API key names.** The output from `supabase start` uses two newer terms: `Publishable` (the key that's safe to ship in the mobile app — used to be called "anon key") and `Secret` (the key that must stay server-side — used to be called "service role key"). Both old and new names work with the JS SDK, but our `.env.example` still uses the old names. Worth a small rename pass before we wire up the client.

**Generated the TypeScript types from the live database.** Ran `npx supabase gen types typescript --local > apps/mobile/types/database.ts`. The output is a 79KB file describing every table, column, enum, and RPC the app can talk to. From here on, the editor will catch typos like a misspelled column name before the code ever runs. The mobile app's typecheck (`tsc --noEmit`) passes silently — nothing else broke.

**Adopted the new Supabase key naming everywhere.** Renamed the env variables to match Supabase's current vocabulary:

- `EXPO_PUBLIC_SUPABASE_ANON_KEY` → `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (safe to ship in the mobile bundle)
- `SUPABASE_SERVICE_ROLE_KEY` → `SUPABASE_SECRET_KEY` (server-only, bypasses Row Level Security)

Updated [`.env.example`](../../.env.example), [`apps/mobile/lib/supabase.ts`](../../apps/mobile/lib/supabase.ts), and the comment in [`scripts/seed-venues.ts`](../../scripts/seed-venues.ts). Also pointed the URL placeholders at the local CLI defaults (`http://127.0.0.1:54321` and the local Postgres on port 54322) so a fresh clone can run `cp .env.example .env`, paste in the two keys from `supabase start`, and the app should boot without further config.

**Wired up the Supabase client.** [`apps/mobile/lib/supabase.ts`](../../apps/mobile/lib/supabase.ts) now exports a typed `supabase` client with:

- the React Native URL polyfill imported (Supabase's HTTP layer uses `URL`, which RN doesn't ship by default),
- AsyncStorage as the session store (so users stay signed in across app launches),
- `autoRefreshToken: true` (background-refreshes the access token before it expires),
- `detectSessionInUrl: false` (we're a native app — magic-link tokens come back through a deeplink, parsed by code we'll add later, not by Supabase scanning the URL bar),
- a startup guard that throws a clear, helpful error if the env vars aren't set, instead of letting the first query fail with a confusing `Failed to fetch`.

`tsc --noEmit` still passes.

## Decisions made

- **`public.users` mirrors `auth.users`** via an `AFTER INSERT` trigger on the auth table. Standard Supabase pattern — gives us a normal table to FK against without modifying the locked-down `auth` schema.
- **`is_admin()` is `security definer`** with `search_path = public`, granted to `anon` and `authenticated`. RLS policies can then call it freely without granting `select` on `staff` to everyone.
- **Verified-owner gate on venue-child writes.** Every write policy on `venues`, `venue_photos`, `opening_hours`, `events`, `venue_categories` requires a row in `venue_owners` *with* `verified_at IS NOT NULL`. A pending claim can read but can't edit. Admins can always write.
- **`plan_items` uses two nullable FKs** (`venue_id`, `event_id`) with a `CHECK ((venue_id IS NOT NULL) <> (event_id IS NOT NULL))` to enforce exactly-one. Cleaner than a polymorphic `target_id + target_type`; the typed FKs let Postgres enforce referential integrity.
- **`subscriptions` has a `CHECK`** that pairs `plan_type` to `venue_id` correctly: Venue Pro must reference a venue, Parent Premium must not. Avoids "orphaned premium" bugs at the database level.
- **`bookings_clicks.user_id` is nullable** with `ON DELETE SET NULL`, so anonymous click attribution still works (e.g. if the booking-redirect Edge Function records a click before the user logs in) and account deletion preserves aggregate stats.
- **`reviews.user_id` is also `ON DELETE SET NULL`**, so deleting an account anonymises their reviews rather than removing them. DESIGN.md §10 said erasure should anonymise reviews — this is the row-level mechanic.
- **`weekday` in `opening_hours` is ISO 1–7** (Mon=1, Sun=7) so it lines up with `extract(isodow from ...)` in the `open_now` filter.
- **`open_now` evaluates against `Europe/London`** time. NI uses Europe/London, ROI uses Europe/Dublin — they're synonymous (same offset, same DST rules), so one named zone works for both.
- **`venues_within` takes `filters jsonb`** rather than a long parameter list. Gives us a stable function signature as we add filters; the cost is a slightly more verbose call site in the app.
- **Default-deny RLS with explicit allow policies** rather than open-by-default with negative policies. Standard Supabase practice and easier to reason about ("can this user do this?" is a positive question, not a "is anything blocking this?" question).
- **Storage buckets are *not* in this migration.** They live in the `storage` schema and are normally created via the Supabase dashboard or `storage.buckets` API. Documented in DATA_MODEL.md so it doesn't get forgotten.
- **Supabase CLI is a project dev dependency**, not a global Homebrew install. Pins the CLI version in `package.json` so `npm install` is enough to get any future contributor onto the same version. The existing `npm run supabase:*` proxy scripts work unchanged because npm puts `node_modules/.bin` on PATH for scripts. Updated [`docs/SETUP.md`](../SETUP.md) and the comment in [`package.json`](../../package.json) to match.
