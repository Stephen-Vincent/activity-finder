# 2026-05-05 — Auth: 6-digit OTP sign-in flow

**Duration:** ~3 hours
**Where I started:** Database schema and typed Supabase client are in place (commit `104ed85`). The mobile app boots but had no sign-in screen and no idea who the user was — every relevant file was a stub returning `null`.
**Where I ended:** A working 6-digit OTP sign-in flow, tested end to end on a real iPhone. Type email → get code in Inbucket → type code → land on home with your email shown → sign out → bounce back to sign-in. Typecheck stays green.

## Goal

Wire up the first auth method. Plan going in was email magic link: type email, tap a link in your inbox, app opens you signed in. End the session with that working on a real device.

## What got done

The plan changed mid-session. Magic links turned out to be brittle in dev (broken default email template, deep-link routing issues, `127.0.0.1` versus LAN IP), so we **pivoted to a 6-digit OTP code flow** halfway through. The user types their email, gets a 6-digit code in their inbox, and types it back into the app. Same `signInWithOtp` Supabase API, different UI and different email template. Magic links can come back later if we want them.

Below is the full journey, including all the surprises along the way.

### The pieces that landed

**A pure deep-link parser.** [`apps/mobile/lib/auth/deeplink.ts`](../../apps/mobile/lib/auth/deeplink.ts) is a single function — `parseAuthDeeplink(url)` — that pulls `access_token` and `refresh_token` out of a Supabase magic-link callback URL fragment. We didn't end up using it for the OTP flow, but it's still wired into the AuthProvider for future Apple Sign In, Google sign-in, or password-reset flows that come back via deep links. Pure function, six smoke-test cases all passing.

**The AuthProvider context.** [`apps/mobile/components/providers/AuthProvider.tsx`](../../apps/mobile/components/providers/AuthProvider.tsx) is the source of truth for "who's signed in?". On mount it (a) asks Supabase for any cached session in AsyncStorage, (b) subscribes to `supabase.auth.onAuthStateChange` for everything that follows, and (c) listens to `expo-linking` events and runs them through the parser, so future deep-link auth flows just work. Exposes `{ session, user, loading, signOut }` via a `useAuth()` hook.

**The sign-in screen.** [`apps/mobile/app/auth/sign-in.tsx`](../../apps/mobile/app/auth/sign-in.tsx) is a single screen with two states: email entry and code entry. The code-entry input has the right keyboard hints (`keyboardType="number-pad"`, `textContentType="oneTimeCode"`) so iOS will offer auto-fill if it picks the code out of an SMS in the future. Both buttons are guarded against double-taps with both a React state (`busy` → spinner + greyed-out style) and a synchronous `useRef` lock that flips on the very first tap, before React's setState has a chance to re-render. The state alone wasn't enough — a fast finger could fire the handler twice in the same animation frame and we'd send two codes, which silently invalidated the first one and confused the user.

**The custom email template.** Supabase's default magic-link template emits a URL with a broken path (`/verify` instead of `/auth/v1/verify`) on this CLI version. Rather than dig into why, we replaced the template with [`supabase/templates/magic_link.html`](../../supabase/templates/magic_link.html) — a minimal HTML email that prominently displays `{{ .Token }}`, the 6-digit OTP code. Wired in via `[auth.email.template.magic_link]` in [`supabase/config.toml`](../../supabase/config.toml).

**Root layout + AuthGate.** [`apps/mobile/app/_layout.tsx`](../../apps/mobile/app/_layout.tsx) wraps the whole app in `SafeAreaProvider` → `GestureHandlerRootView` → `AuthProvider` → `AuthGate` → `Stack`. The `AuthGate` component is the one we're proudest of — see "Decisions made" below.

**Route gating in `app/index.tsx`.** Reads `useAuth()` and redirects to `/auth/sign-in` (no session) or `/(tabs)/home` (session). With `AuthGate` now in place this is technically redundant — `AuthGate` will catch the same cases — but leaving it as a fast-path means we don't render `null` for an extra frame on cold start when the session is already cached.

**Minimal tabs to land on.** The redirect target needed something to render or you'd hit a blank screen. So `(tabs)/_layout.tsx` is now a real `<Tabs>` with four named tabs (Home, Discover, Plans, Profile), no icons yet, and `(tabs)/home.tsx` is a placeholder showing "You're signed in" + the user's email + a Sign out button. Sign out has the same double-tap guard as sign-in.

**Typecheck stays green** through every iteration.

### The surprises (and how we got around each one)

**Two stale Expo plugins crashed the bundler.** First time we ran `npm run start`, Expo's config plugins for Stripe and Mapbox both blew up because we haven't actually configured either of them yet (`Cannot read properties of undefined (reading 'merchantIdentifier')`). They were listed in `app.json`'s `plugins` array but had no config. Removed them from `plugins`. The npm packages are still installed — when we eventually wire up Stripe and Mapbox we'll add the entries back with proper config.

**React version mismatch (19.2.5 vs 19.1.0).** After the plugins were unblocked, the app on the phone showed a red screen: `Incompatible React versions: react: 19.2.5, react-native-renderer: 19.1.0`. The renderer is bundled inside `react-native@0.81.5` and is fixed at 19.1.0; somehow `npm install` had hoisted `react@19.2.5` despite our `package.json` pinning `react: "19.1.0"`. `npx expo install --fix` thought 19.2.5 was correct for SDK 54 and refused to fix it. We added an `overrides` block to the root `package.json` to pin react to 19.1.0 hard, blew away `node_modules` and `package-lock.json`, reinstalled. Sanity check: `node -p "require('./node_modules/react/package.json').version"` → `19.1.0`. Bundled clean.

**`127.0.0.1` doesn't work from the phone.** The first sign-in attempt failed with `Network request failed`. On the phone, `127.0.0.1` means the phone — not the laptop. The fix was to change `EXPO_PUBLIC_SUPABASE_URL` in `apps/mobile/.env` from `http://127.0.0.1:54321` to `http://192.168.1.136:54321` (the laptop's LAN IP, which Metro had helpfully printed). This change stays in the local `.env` only — the committed `.env.example` keeps `127.0.0.1` for new contributors. If the laptop's IP changes (Wi-Fi switch, router restart) this needs updating. Not painful enough yet to automate.

**Magic-link template path was wrong in the email.** The default email Supabase generated for `signInWithOtp` pointed at `http://127.0.0.1:54321/verify?...` — but the API gateway exposes the verify endpoint at `/auth/v1/verify`, not `/verify`. Manually inserting `/auth/v1/` made the verify endpoint return a redirect to `activityfinder://#access_token=...`, which the laptop browser couldn't open. This was the trigger for pivoting to the OTP code flow instead of fixing the magic-link template.

**OTP `content_path` is relative to where the CLI runs.** First attempt at the email-template config used `content_path = "./templates/magic_link.html"`, which works when the CLI is invoked from inside `supabase/`. Our `npm run supabase:*` scripts run from the project root, so the path needed to be `supabase/templates/magic_link.html`. Without that prefix the CLI fails with a confusing `no such file or directory` message at start-up. Documented in a comment so future-us doesn't trip over it again.

**OTP expiry default was too short.** Even with the correct template and a fresh code, the verify call kept returning "token has expired". On a recent CLI version the default email-OTP expiry is short enough that switching from the app to Inbucket and typing six digits is sometimes enough to burn through it. Set `otp_expiry = 600` (10 minutes) explicitly in `[auth.email]` in `config.toml`. The hidden second cause was a fast double-tap on "Send sign-in code" creating two emails — the first invalidated by the second — and the user reaching for the first.

**Sign-in worked but didn't navigate.** With everything else fixed, the OTP exchange succeeded but the app stayed on the sign-in screen until a manual reload. Same problem on sign-out — spinner stuck, no navigation. The redirect logic in `app/index.tsx` only fires when the user is on `/`. Once you're on `/auth/sign-in` or `/(tabs)/home`, that file isn't mounted, so it can't react to a sign-in or sign-out. Fix: a small `AuthGate` component that lives inside the navigator (alongside `<Stack>`), uses `useSegments()` to know which group you're in, watches the session, and `router.replace()`s when the two disagree. Standard Expo Router auth pattern.

## Decisions made

- **OTP code, not magic link.** Magic links chained too many things that all had to work together: the email template URL had to be right, the host had to be reachable from the phone, the deep-link round trip had to survive iOS Mail → Safari → Expo Go → the app. The OTP code flow is a single API call with no URL plumbing. Same flow on simulator, real device, Expo Go, dev build, production. Magic links can be added back as an option later — most users won't notice the difference, but if we want one-tap parity with Apple's "Sign in with Apple" we'll come back to this.
- **Synchronous `useRef` guard alongside the React state for double-tap protection.** The visible busy/disabled state still does its job, but React's setState is asynchronous, so a fast finger can fire the handler twice before the re-render. The ref flips on the first tap and any further taps see `inFlight.current === true` and bail before any side effects run. Both the sign-in and sign-out buttons use this pattern.
- **`AuthGate` lives inside the Stack, not above it.** The Stack itself is what owns the segments and the router. Anything that needs to read the current route and call `router.replace()` has to be a descendant of Stack. So `AuthGate` is a thin component sitting *between* `AuthProvider` and `<Stack>`.
- **Pin `react@19.1.0` via the root `package.json` `overrides` block.** `npx expo install --fix` is opinionated about what react version SDK 54 wants (19.2.5) but the renderer bundled inside `react-native@0.81.5` is fixed at 19.1.0. Hooks fail at render time if these don't match exactly. The override is the simplest hammer that wins the argument.
- **Custom email template, even though it duplicates effort we'd have to redo at production.** The default template emits a broken URL on this CLI version, and we couldn't easily diagnose why. Writing our own is two minutes' work and gives us full control.
- **Set `otp_expiry = 600` explicitly.** Don't rely on the CLI's default — it's been moving and 60 seconds is too short for a real human to switch context to their email and back.
- **Removed the temp `redirectTo` console.log** before committing — the OTP flow doesn't use it, so it'd just be noise.
- **Stripe and Mapbox plugins removed from `app.json`'s `plugins` array.** Add them back when we actually integrate those products. The npm packages stay installed; nothing is broken at code level.

## What's deferred / open

- **Magic links.** We can add them back as a second option on the sign-in screen later. Lower priority than getting the rest of the app shippable.
- **Apple Sign In + Google.** Needed for TestFlight prep when we add any social login. The deep-link plumbing in `AuthProvider` is already there waiting for them.
- **Account deletion.** Required by App Store policy if we offer account creation. Not in this session.
- **Real toast / error UI.** Errors currently show as inline red text near the button. Once we have a toast system, route them through that.
- **Onboarding redirect.** `app/index.tsx` and `AuthGate` should bounce signed-in users without a profile or children to `/onboarding`, not straight to `/(tabs)/home`. That hop is wired up as soon as `onboarding.tsx` is built.
- **The `lib/auth/deeplink.ts` parser is unused right now.** Kept for the future Apple / Google / password-reset flows. If we decide we'll never need it, delete and drop the AuthProvider listener.
- **`assets/splash.png` is missing,** producing a cosmetic warning at boot. Either rename `splash-icon.png` or update `app.json`. Not urgent.
- **The Supabase URL in `.env` is hard-coded to my LAN IP.** Will need updating when I switch networks. Could auto-detect via `expo-constants` later.
- **Sign-out lives on Home, not Profile.** It's the wrong place long-term — leaving it there for now so testing is fast. Move when Profile is built.

## Where to pick up

The plumbing is now in place. The natural next step is **the onboarding screen** at `apps/mobile/app/onboarding.tsx` — a step-by-step flow that captures home location (postcode/Eircode → geocoded via Mapbox), then adds the user's children (nickname + DOB), then permission prompts. That writes the first real rows to `users` and `children`, and it's the first chance to use the typed Supabase client for a real query.

If onboarding feels too big for one sitting, two smaller alternatives:

1. **Build the Profile tab.** Move the sign-out button there. List the user's children (none yet, but the empty state matters). Add a "delete my account" placeholder.
2. **Wire up Mapbox in the Discover tab.** A blank map at the user's current location is a small but motivating win, and it forces us to actually configure the Mapbox plugin we removed from `app.json`.
