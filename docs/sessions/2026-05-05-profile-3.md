## Goal

Build the Profile tab. Show everything we know about the user, let them change most of it inline, give children full CRUD (with auto-tracked age), and provide a clean home for sign-out and the future "delete account" button.

## Scope decisions before starting

- **Editable: display name, marketing opt-in, country, postcode (re-geocoded), children (rename + change DOB).**
- **Read-only: email.** Changing email needs the Supabase email-change flow; left as a "Change email — coming soon" disabled row so the user knows it exists.
- **Account deletion: placeholder.** Tapping it shows a "Coming soon" alert. Wiring it up properly needs a SQL migration (`SECURITY DEFINER` RPC to delete `auth.users`) and a `supabase db reset`, deferred to its own session.
- **Children's age tracks automatically.** The DB stores `dob` only. `computeAgeYears(dob, today)` runs on every render, so a child who's "4" now will be "5" the morning after their birthday with no DB writes.

## What got done

**The Profile tab.** [`apps/mobile/app/(tabs)/profile.tsx`](../../apps/mobile/app/(tabs)/profile.tsx) renders four sections: About you, Where you are, Your kids, Account. Each section owns its own edit state.

- *About you*: display name (inline Edit / Save / Cancel), email (read-only with a "Change email — coming soon" caption), occasional-emails toggle (saves immediately on flip — no edit/save dance for a binary).
- *Where you are*: country + postcode, view-mode collapsed to two lines, edit-mode expands into a country picker + postcode input + Mapbox Check + Save/Cancel. Cancel restores the original state.
- *Your kids*: list of children with computed age, each with Edit / Remove. Remove pops a destructive confirmation alert. + Add inline.
- *Account*: Sign out (with spinner + double-tap guard), and a Delete-account button that pops a "Coming soon" alert.

**`updateChild` joined `useChildren`.** New mutation in [`apps/mobile/hooks/useChildren.ts`](../../apps/mobile/hooks/useChildren.ts) that PATCHes only the keys the caller actually set. The Supabase typed update needed an explicit `{ nickname?, dob?, notes? }` type so it didn't fall through to "any column at all" — caught at typecheck time.

**`updateProfile` joined `useAuth`.** [`apps/mobile/components/providers/AuthProvider.tsx`](../../apps/mobile/components/providers/AuthProvider.tsx) now exposes `updateProfile(patch)`. It UPDATEs `public.users` and re-reads the row in one call so consumers don't need to know which fields are cached. Returns `{ error: string | null }`. Used everywhere in Profile that writes profile fields.

**Sign-out moved off Home.** Home is now a thin placeholder with greeting + postcode and no actions. The sign-out button, with its double-tap guard and spinner, lives in Profile's Account section where it belongs long-term.

**A round of refactor and de-duplication.** profile.tsx grew to ~700 lines and a chunk of it duplicated onboarding.tsx — same DOB inputs, same country picker card, same child add form. Extracted the shared bits to standalone components and a small lib helper, and pointed both screens at them:

- [`components/Section.tsx`](../../apps/mobile/components/Section.tsx) — iOS-Settings "label above white rounded card".
- [`components/forms/CountryChoice.tsx`](../../apps/mobile/components/forms/CountryChoice.tsx) — country option card; optional `selected` prop so onboarding (no selection state) and Profile (highlight current pick) both use it.
- [`components/forms/DobInputs.tsx`](../../apps/mobile/components/forms/DobInputs.tsx) — three numeric fields for DD/MM/YYYY.
- [`components/children/ChildAddForm.tsx`](../../apps/mobile/components/children/ChildAddForm.tsx) — name + DOB + Save/Cancel, owns its own validation and busy state. Used by onboarding's children-add step and Profile's "+ Add" affordance.
- [`components/children/ChildRow.tsx`](../../apps/mobile/components/children/ChildRow.tsx) — display + inline edit. Profile-only for now but big enough to deserve its own file.
- [`lib/dob.ts`](../../apps/mobile/lib/dob.ts) — single `validateDob(day, month, year)` helper that both forms call.

Numbers: profile.tsx went from ~700 → 587, onboarding.tsx from 753 → 576, six new files at 38–193 lines each. Total lines up slightly because each new file has its own header/styles, but every change to DOB validation, country card style, or child-row layout now happens in one place.

**Typecheck stays green** all the way through.

### Late additions (same session)

After the first round was working we made two more changes:

**Sign-out is on Home AND Profile.** The user explicitly preferred sign-out staying on the Home tab, since that's where they land first when signed in. Kept it on Profile too because long-term that's where it semantically belongs (alongside delete-account). Both buttons share the same double-tap guard and spinner pattern.

**"Use my current location" button in onboarding and profile.** New affordance: tap a button, the app asks for foreground location permission, gets a single coordinate fix, reverse-geocodes it via Mapbox, decides whether the result is on the island of Ireland, and either fills in country + postcode (and shows the confirmation card so the user can sanity-check) or pops a polite "we only cover Ireland for now" alert.

The pieces:

- [`apps/mobile/lib/location.ts`](../../apps/mobile/lib/location.ts) — thin wrapper around `expo-location`. Permission + coordinate fix + a typed failure result (`'denied' | 'unavailable' | 'error'`).
- `reverseGeocode(lng, lat)` and `classifyLocation(country, postcode)` added to [`apps/mobile/lib/mapbox.ts`](../../apps/mobile/lib/mapbox.ts). Classification rule: `country=ie` → ROI, `country=gb` + postcode starts with `BT` → NI, anything else → off-island.
- [`apps/mobile/components/forms/UseLocationButton.tsx`](../../apps/mobile/components/forms/UseLocationButton.tsx) — the button + the orchestration. Three callbacks (`onResult`, `onOffIsland`, `onError`) so the parent decides what to do with each outcome.
- Wired into the postcode step in onboarding and the WhereYouAre edit form in Profile, both with the same off-island alert text.

Both flows tested on a real iPhone in Belfast: permission prompt fires, coords come back, postcode populates, country auto-corrects if it differs from what the user picked. The off-island branch wasn't exercised in practice (no easy way to fake GPS on a real device) — the logic is small enough to trust on inspection but flagging here so a future-us knows it's "probably correct" rather than "verified on device".

## Decisions made

- **Marketing toggle saves immediately, display name uses Edit/Save/Cancel.** Toggles are quick to revert — making them save-on-flip matches iOS Settings and avoids a tedious save dance. Free-text fields have a real "did I get this right?" moment, hence the explicit save.
- **`updateProfile` lives on `useAuth`, not as a separate hook.** Profile edits are a property of the auth session — same place we already keep `signOut` and `refreshProfile`. Adding a `useProfile` hook would force every consumer to subscribe to two things that always change together.
- **`updateChild` PATCHes only set keys, never clobbers others.** The Supabase strict typed update would happily wipe a column if you passed `undefined` — building a `{ nickname?, dob?, notes? }` patch object explicitly avoids that. Caught at typecheck time the first time around.
- **Postcode in Profile re-uses the onboarding flow shape.** Country picker + postcode + Check + Save. We considered a one-shot "type and tap save" but separating the geocode confirmation from the save gives the user a chance to back out if Mapbox interpreted "BT1 1AA" as somewhere unexpected.
- **Refactored to extract shared components after the screen worked, not before.** Wrote profile.tsx as one big file first, confirmed it worked end-to-end, then pulled the shared bits out. Lower risk than refactoring a stub.
- **Each shared component owns its own styles file rather than importing from a shared theme.** We don't have a design system yet; pulling out a `theme.ts` would be premature. Worth revisiting once we have 4-5 more screens.
- **Sign-out lives on both Home and Profile.** Home for the "I'm done, get me out of here" reflex; Profile because that's the long-term semantic home.
- **"Use my current location" auto-corrects the country.** If the user picked NI in step 2 but the GPS says they're in Dublin, the location flow silently switches to ROI rather than throwing an error. They picked the wrong country accidentally; the app shouldn't make them go back.
- **Off-island branch is an alert, not a redirect.** We could refuse to save and force the user back, but the polite alert + manual-entry fallback feels more respectful: if a tourist wants to look up Belfast soft-plays from London, that's a valid use case.

## What's deferred / open

### Must implement before ship

- **Email-change flow.** Use Supabase's built-in `supabase.auth.updateUser({ email })` which sends a 6-digit code to the new address. Without this, users who lose access to their old inbox have no recovery path that preserves their data. App Store reviewers also flag this as user-hostile. Replace the "Change email — coming soon" row with a real flow.
- **Account deletion (real).** Right now the button shows an alert. To actually delete:
  1. Add `supabase/migrations/0002_delete_account.sql` with a `SECURITY DEFINER` RPC `public.delete_account()` that `delete from auth.users where id = auth.uid()`. The cascades clean everything else.
  2. Mobile-side: call `supabase.rpc('delete_account')`, then `supabase.auth.signOut()`. AuthGate will route to sign-in.
  3. Confirmation dialog with strong language (the action is irreversible).
  4. App Store policy requires this for any app that allows account creation.

### Other deferred

- **Avatar upload.** Schema has `avatar_url`; UI has nothing. Add when we wire up Supabase Storage.
- **Preferences section** (currency display, distance unit, notification settings).
- **Premium / subscription view** — needs Stripe wired up first.
- **Venue-owner section** — only relevant once we have venue claim approvals.
- **Legal links** (privacy, terms, contact) — placeholder rows when we have the URLs.
- **Native date picker** for child DOB. The three-field input works but isn't as smooth as `@react-native-community/datetimepicker`.
- **A real toast / inline-error system.** The current inline-red-text pattern works but is inconsistent across screens.
- **Retry/backoff on profile-fetch failure.** AuthProvider currently logs and leaves `profile` null; AuthGate then waits indefinitely. Acceptable for v0, not for production.
- **Off-island flow is unverified on a device.** The classifier is small and was derived from Mapbox's response shape, but we never actually triggered the "you're in London, please use manual entry" path on hardware. Worth a deliberate test trip — or a small debug screen for fake coords — before a real user trips it.

## Where to pick up

1. **Discover tab + Mapbox map.** The most visible and motivating feature. Means standing up an Expo development build (Mapbox's native SDK doesn't run in Expo Go), wiring up `@rnmapbox/maps`' Expo plugin properly with the secret token, and dealing with the LAN-IP issue again. Bigger investment but a huge "the app is real" moment when the map renders.

I'd lean **seed venues first** — it's a smaller session, no native build complications, and means when we do bring the map up there are actual dots to see. Discover with no data is just a blue rectangle.
